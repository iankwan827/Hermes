- [Hermes配置] Token Plan的vision_analyze有api-key header丢失的bug — `vision_analyze`工具在Xiaomi Token Plan上请求时丢失api-key header导致失败
  - **解决方案**：将config.yaml中`auxiliary.vision`改为按量付费的`xiaomi-regular` provider（不是Token Plan的provider），同时配置fallback也为xiaomi-regular

  - **解决方案**：使用`hermes fallback add`交互式命令选择provider编号（如选38=xiaomi-regular），不能手动编辑config.yaml的fallback字段
  - 来源：CLI session 20260609_105827

## 通用经验教训（持续积累）

### 工具使用铁律

1. **vision_analyze现在可用** — Xiaomi Token Plan key配好后（2026-06-09实测通过），api-key header问题已修复。直接用vision_analyze即可，不用再绕analyze-image skill
2. **Python脚本用uv python** — 系统python不可用，路径：`E:\Users\Administrator\AppData\Roaming\uv\python\cpython-3.11-windows-x86_64-none\python.exe`
3. **写Python脚本不要用 `write_file`** — 会破坏f-string语法，用terminal heredoc
4. **修改脚本后必须用 `python -m py_compile` 验证** — write_file和skill_manage都可能破坏字符串
5. **.env文件需手动加载** — Python脚本直接运行拿不到环境变量，必须先读取 `~/AppData/Local/hermes/.env`
6. **OpenCLI ref容易stale** — 每次操作前必须重新获取最新ref
7. **fallback配置用 `hermes fallback add`** — 不能手动编辑config.yaml的fallback字段，要用交互式命令选择provider编号

### 流程管理

1. **自动化能力边界要诚实** — 不要过度承诺自动化率，实际做不到的就标0%
2. **用户说"先跑"就先跑** — 不要反复确认
3. **交付必须完整** — 不接受"大部分完成+几个失败"
4. **遇到问题先搜社区** — B站视频优先，不要直接套通用方案
5. **热点事件时间线必须核实** — 不能凭记忆写综艺赛程、新闻事件等时效性内容
6. **用户更信任社区搜索而非AI建议** — 崩坏崩溃问题用户自己搜索B站就解决了，AI建议的"重装驱动"方案反而复杂
7. **skill精炼需求** — 用户希望从线上课笔记直接提炼成可执行skill，比knowledge-extraction更注重可操作性

### 文件处理

1. **长图必须切割** — 高宽比>3或高度>8000px的图片，整张发送会严重截断内容
2. **超大图先缩小** — 20M+像素用PIL缩放后再处理，否则MemoryError
3. **图片尺寸必须用PIL读取** — 不能手动解析文件头（只对PNG有效，JPEG会出错）
4. **批量处理前先分析源文件夹结构** — 不要直接跑脚本
5. **0字节文件必须主动重试** — 不要只记日志就完事
