# dbs-video-workflow

**版本**：v1.0  
**创建日期**：2026-05-03  
**适用场景**：把一个想法/选题/产品，系统化地做成一条短视频的完整文案

---

## 核心逻辑

视频制作不是线性的"写稿→拍摄"，而是 **4 个心智阶段，每个阶段解决一个灵魂问题**：

```
阶段 1｜吃透选题
  灵魂问：你真的懂这个东西为什么存在吗？
  输出：选题的底层情绪 + 赛道定位 + 90% 痛点清单 + 数据支撑

阶段 2｜钩住人群
  灵魂问：3-90 岁的人凭什么点进来？
  输出：10+ 条爆款开头方案 + 痛点全覆盖清单

阶段 3｜承诺蜕变
  灵魂问：看完我这 5 分钟，用户从 A 变成什么 B？
  输出：A→B 路径（低能量 → 跃迁词 → 高能量 → 期许）+ 加过心理学/底层逻辑的 800 字

阶段 4｜聊透输出
  灵魂问：你能不拿稿子对着镜头说完吗？
  输出：内化后的大框架 + 删完废话的终稿
```

---

## 审核步骤（Phase 4之后必须执行）

⚠️ **写完终稿后，必须自动跑一遍 `video-content-audit` skill 审核，不要等用户提醒。** 用户原话："你没跑skill啊"——审核是内建流程，不是可选步骤。

审核通过后再呈现给用户。如果审核不通过，先修复再呈现。

## 栋笃笑叙事结构（黄子华风格）

用户账号定位粤语栋笃笑/吐槽，写稿时优先使用黄子华式对话叙事：

**标准结构：**
```
爆款开头（1-2句，定话题）
  ↓
人物故事铺垫（"去年我同阿强讲……"）
  ↓
现实对照（"今日见到金价跌咗400"）
  ↓
炫耀→反转（"好彩我叫你卖晒" → "我冇卖啊"）
  ↓
吐槽人物 → 顺理成章引出专业知识
  ↓
金句收尾
  ↓
反问互动（要跟故事呼应，不要泛泛而谈）
```

**人物角色使用规则：**
- 阿强/具体人物 = 观众的代言人，用来制造笑点和共鸣
- 人物可以蠢、可以不信你、可以倒霉——这些都是笑点来源
- 专业知识由"我"输出，人物负责挨骂/被吐槽
- 结尾反问要跟人物故事呼应（"你啲金卖咗未啊？"✓ "你信唔信五行？"✗）
- **⚠️ 阿强叙事可以用单方面叙述（2026-06-25 用户纠正）：** 不一定要用"我话...佢话..."对话体。单方面讲故事也可以："我有个朋友阿强，前几日喺群里话...我问佢讲边个，佢话...佢唔出声。" 用户原话："不需要用对话，好像刚刚那样，当方面讲阿强故事就可以了"。选择对话体还是叙述体取决于内容——喷人用叙述体更狠，共鸣用对话体更自然。

**开头两段式结构：**
1. 第一段：爆款开头（用 /开头 skill 的模板套用，1-2句，定话题+制造悬念）——这必须是文案的第一句话，不是单独的标题元素
2. 第二段：人物叙事（阿强故事，用对话体展开）
- 两段之间自然过渡，不要生硬

**⚠️ 开头第一句就是选的那句（2026-06-12用户纠正）**：
- 用户选了"这个世界上只有两种人..."作为开头，这就是文案的第一句话
- 不要另外加一个标题行，不要把选的开头和正文分开
- 用户原话："开头第一句肯定是我们选的那句选题啊"

**结构性回声技巧（2026-06-10 新增）：**
开头抛出的关键词/数字，在正文中间再次呼应，形成结构闭环。

**钩子密度要求（2026-06-10 用户纠正）：**
每30-45秒必须有一个留人钩子，不能有超过40秒的"平铺区"。

## Pitfalls

### 工具使用

**⚠️ 4个阶段必须完整跑完，不能跳步（2026-06-21 用户纠正）：** 用户原话："dbs的几个阶段你都没跑完吧"。每个阶段必须输出对应产物后才能进入下一阶段。阶段1必须输出90%痛点清单，阶段2必须输出10+条开头方案，阶段3必须输出800字初稿，阶段4才输出终稿。不能把阶段2-4压缩成一步直接给终稿。

**⚠️ 每个阶段产物必须写入文件，不能只在聊天里写（2026-06-24 用户纠正）：** 用户原话："咋感觉你忽悠我呢，我都没在cli的输出看到你跑的4阶段"。只在聊天回复里写摘要不算跑完，必须用 write_file 把产物存到 `D:/hermes-agent/文案/2026-MM-DD_主题_阶段N.md`。用户要能从文件系统看到实际产出。

**⚠️ 开始工作流时立刻加载 huang-zihua-perspective（2026-06-24 用户纠正）：** 用户问"你有用黄子华的skill吗"——工作流阶段2明确要求用黄子华模板，但agent经常忘记加载。正确做法：在阶段1完成后、阶段2开始前，必须先 `skill_view(name='huang-zihua-perspective')`。不要等到写完文案才想起来。

**⚠️ OpenCLI用Chrome不是Camofox（2026-06-20 用户纠正）：** OpenCLI复用的是本机Chrome浏览器的登录态，不是Camofox。用户原话："你是不是忘记了opencli，他是用chrome浏览器"。Camofox是独立的无头浏览器，需要单独启动。OpenCLI直接用已打开的Chrome。

**⚠️ 搜标签不要用OpenCLI hashtag search（2026-06-20 用户纠正）：** `opencli douyin hashtag search` 走的是 creator.douyin.com API，不是抖音主页。正确做法：用 `opencli browser default open "https://www.douyin.com/search/关键词"` → `opencli browser default state` 看结果 → 点进视频看标签。

**⚠️ 小红书 note/comments 命令必须传完整签名URL（2026-06-23）：** `opencli xiaohongshu note <note-id>` 和 `opencli xiaohongshu comments <note-id>` 现在要求完整URL（含xsec_token），不能只传note ID。正确做法：从搜索结果的 `url` 字段取完整链接传入。示例：`opencli xiaohongshu note "https://www.xiaohongshu.com/search_result/xxx?xsec_token=yyy=&xsec_source=" -f plain`

**⚠️ 搜抖音热点/话题：优先用eval提取文本，vision_analyze作为补充（2026-06-22 更新）：**
- eval提取文本更快更稳定：`opencli browser default eval "document.body.innerText"`
- vision_analyze现在可用（config已加 `agent.image_input_mode: text`），但只支持本地文件路径截图，不支持远程URL
- vision_analyze适合分析搜索结果页面的整体布局和视频封面，eval适合提取具体文字内容
- 两者配合使用：先eval拿文字数据，需要看视觉元素时再截图+vision_analyze
抖音热搜API（免登录）：`curl -s "https://www.douyin.com/aweme/v1/web/hot/search/list/" -H "User-Agent: Mozilla/5.0"`

### 文案写作

**⚠️ 解释要简单直接，不要用专业术语（2026-06-20 用户纠正）：** 写科普/辟谣类文案时，解释要一句话讲完，不要用化学式、公式等专业术语。用户纠正："这句不需要科普，直接说甲醛没了碳就剩下水就好了"。❌ "甲醛的化学式 CH₂O，脱碳即系掹走个碳原子，剩低 H₂O——水" ✅ "甲醛冇咗个碳，咪剩返水咯"。原则：观众是来看吐槽的，不是来上课的。

**⚠️ 阿强人设可以升级（2026-06-20 用户纠正）：** 阿强的学历/背景可以设定得更高，反差更大。用户说："阿强大学毕业不更好吗"——大学毕业都被骗，比"读过书"更荒谬。原则：阿强越"正常"/"有文化"，被骗的笑点越强。

**⚠️ 粤语用词要精准（2026-06-20 用户纠正）：** 粤语近义词要选最精准的那个。用户改"丢架"为"落面"——"丢架"是自己丢脸，"落面"是被人落脸（孩子当众拆穿爸爸）。在这个场景里，是孩子"落"爸爸的面，不是爸爸自己"丢"架。原则：写完每个粤语词问自己——主语是谁？是自己做的还是别人做的？

**⚠️ 不要用普通话词冒充粤语（2026-06-25 用户纠正）：** "怂"是普通话，粤语没有这个词。用户纠正："粤语没有最怂这个词把"。粤语用"冇胆"或"缩骨"。常见普通话词误用：怂→冇胆、厉害→犀利、怎么→点解、什么→乜嘢、不要→咪、觉得→认为（粤语用"觉得"）。写完必须逐句检查有没有普通话词混入。

### 转场

**⚠️ 转场不能硬切，要"滑过去"（2026-06-20 用户纠正）：** 文案各段之间不能直接跳到下一个话题。正确做法：永远用上一个故事的最后一个词/情绪做桥梁。最常用的三板斧："问题係""但係""咁即係"。转场技巧详见 huang-zihua-perspective skill 的 references/transitions.md。

### 选题来源

**⚠️ 工作流可以从热搜开始，也可以从研究开始（2026-06-23）：** 不是每次都要先看热搜。用户说"看看今天热点"→没有合适的→转去小红书搜索共鸣类话题→找到具体帖子→深入看评论区和相关内容→提炼选题。这条路径同样有效。关键是：选题来自真实共鸣，不一定来自热搜。

**⚠️ 用户问"具体事情不"是在追问真实案例（2026-06-23）：** 用户看到一个概括性话题（如"35岁硬撑"）后问"有具体事情不"，意思是"这个话题背后有什么真实故事/场景可以展开"。正确做法：1) 看原帖内容 2) 翻评论区找真实故事 3) 搜相关帖子找具体案例 4) 把具体场景列给用户选。不要只给概括性描述。

**⚠️ 小红书评论区是选题素材金矿（2026-06-23）：** 很多帖子内容本身很笼统（一堆感慨），但评论区有大量真实故事（"35想离职又怕找不到工作""合伙人坑了负债百万"）。这些具体场景才是栋笃笑文案的素材来源。研究选题时必须翻评论区。

**⚠️ 找选题要多角度搜索（2026-06-23）：** 用OpenCLI搜小红书时，不要只搜一个关键词。同一个方向试不同角度：如"打工人崩溃"、"成年人崩溃"、"社交恐惧尴尬"、"半熟人尴尬"、"工作群吐槽"——每个角度出来的结果不同，综合看才能找到最适合栋笃笑的具体场景。搜索命令：`opencli xiaohongshu search "关键词" --limit 5 -f json`

### 选题分析

**⚠️ 用户分享真实场景时，先找通用角度，不找领域角度（2026-06-25 用户纠正）：** 用户分享一个具体场景（如八字群里某人的行为）时，不要立刻陷入领域细节（分析八字、十神、风水等）。用户要的是"这个场景背后的人类共性行为"，不是领域分析。用户原话："不是啊，没人让你说八字啊，但是群里总有这么一个人喜欢吐槽"——意思是：先看到"群里怼人不认"这个通用现象，再考虑是否做成视频。正确做法：1) 先识别场景中的通用人类行为 2) 问用户"要不要做成视频" 3) 如果要，再走dbs-video-workflow。❌ 错误：用户分享群聊 → 我立刻分析八字格局、十神性格 ✅ 正确：用户分享群聊 → 我识别出"群里怼人不认"这个通用现象 → 问用户要不要做

**⚠️ 领域专业知识要融入叙事，不要单独科普（2026-06-25 用户纠正）：** 当选题涉及领域知识（如八字十神、心理学概念等）时，把专业知识转化为"对人类行为的观察"融入文案，不要单独开一段科普。用户原话："最好你可以写几句连贯的"——意思是把伤官性格特征写成连贯的观察段落，不是列个清单。❌ 错误："伤官是八字十神之一，代表XX、XX、XX"（科普体） ✅ 正确："伤官的人有个特点，特别喜欢点评别人。他看谁都觉得不够好，看什么事都想插两句。但是他不敢认，因为他自己也心虚。"（观察体）原则：专业知识是"观察工具"，不是"教学内容"。用它来解释人的行为，不是解释术语本身。

**⚠️ 文案篇幅要匹配内容深度（2026-06-25 用户纠正）：** 用户反馈"你这写的，也没几秒就完事了"——说明当话题需要展开分析（如性格特征、行为模式）时，150-250字太短。实际被接受的文案范围应根据内容调整：简单吐槽150-250字，需要分析/展开的话题300-400字。原则：核心洞察保持精炼，但展开分析可以适当加长。用户要的是"完整说完一件事"，不是"越短越好"。

**⚠️ Phase 1分析不要假设受众认知（2026-06-20 用户纠正）：** 分析选题时，不要假设受众知道或不知道某件事。用户纠正："他们不知道是水，所以吓到不敢喝水是不存在的"——受众根本不知道脱碳甲醛=水，他们的反应是"不敢买菜"不是"不敢喝水"。正确做法：先问自己"受众实际上知道什么、不知道什么"，再推导他们的反应。

**⚠️ 伪科普/谣言类选题角度：吐槽信息源，不吐槽主题（2026-06-20 用户纠正）：** 写"伪博主贩卖焦虑"这类选题时，角度应该聚焦在"博主有多蠢/多坏"，不是"这个东西本身怎样"。用户纠正："我们角度是吐槽那些不懂装懂的博主，不是吐槽这个水"。A→B路径：A是"被博主骗了"，B是"发现博主才是傻的"。

### 用户编辑

**⚠️ 用户改完文案就是终版（2026-06-16 用户纠正）：** 用户自己编辑文案时会删掉某些段落/句子，这些删除是有意的。不要在后续修改中把用户删掉的内容加回来。用户原话："你这不是把我删掉不要的又加回来了"。正确做法：先读用户改完的版本，以用户版本为准。

**⚠️ 用户说改就直接改（2026-06-11 用户纠正）：** 用户说"改一下"或指定具体改法时，直接改，不要先分析为什么、给多个选项。用户说"你就直接改字就好了"。

**⚠️ 用户给的结构框架就是终稿结构（2026-06-23 用户纠正）：** 用户说"开头→阿强离职→反转→举例→结尾"时，这就是文案的结构，不要自作主张加段落或重新组织。用户原话："我这句就是一句话，你别拆开说"。用户给的punchline是一句话就保持一句话，不要拆成多段铺垫再给结论。

**⚠️ 不要自己加"尾巴"（2026-06-16 用户纠正）：** 只写用户明确要求的内容，不要自作主张"完善"文案。如果觉得需要加，先问用户。

**⚠️ 小改动不用调工具（2026-06-23 用户纠正）：** 合并一行、改个标点这种微操，记住就好，不要每次都调patch工具浪费token。用户原话："你能不能别浪费token，去改一个句号"。

## 内容角度模式库

用户在选题讨论中反复出现的喜剧模式，可作为选题分析的参考框架：

### 认知差模式（夏虫语冰）

核心笑点：你说的是事实，但对方的认知里根本没有这个东西。不是吵架，是频道不同。

典型场景：
- 你跟HR讲劳动法，HR说"我们公司没这个规定"
- 你跟老板说N+1，老板说"什么N？哪有这个说法"
- 你越解释，对方越觉得你在胡扯

笑点来源：绝望感——不是你不讲道理，是对方根本不在你的频道上。你说的是常识，对方以为你在编。

### 博主不接地气模式（吐槽教的人）

核心笑点：教你的人和你面对的人不在同一个世界。博主教的前提是对方是正常人，但你遇到的根本不是。

典型场景：
- 律师教"你要坚定告诉HR，违法解除是2N"——你照做了，HR说"什么2N？"
- 情感博主教"你要跟伴侣好好沟通"——你伴侣根本不想沟通
- 理财博主教"你要分散投资"——你连本金都没有

笑点来源：荒诞——教你的人按"对方懂"来教，你按"对方懂"去说，但对方是个完全不在这个频道上的人。回头看那个教你的人，他才是傻的。

### 义工/阴公模式（打工倒贴反转）

核心笑点：用"义工"（冇收入但人人赞）对比"打工倒贴"（有收入但全家贴钱），推翻"冇收入=冇价值"的等号。

典型句式：
- "做义工都冇收入，冇人嫌佢。打工打到倒贴嘅，先至阴公。"
- "打工无钱收，最多咪当做义工，但你打工打到要全屋贴钱，简直系阴公，还不如辞工，反而可能会迈向成功。"

适用场景：收入焦虑、职场吐槽、裸辞话题
结构：开头共鸣（冇收入的罪恶感）→ 反转（做义工vs打工倒贴）→ 点题（冇收入唔等于冇价值）

### 通用规律

最好的吐槽类笑点往往来自**信息不对称的绝望感**：
- 不是"我不知道"→ 可以学
- 是"我怎么解释你都不信"→ 绝望
- 不是"你说的不对"→ 可以辩论
- 是"你根本听不懂我在说什么"→ 夏虫语冰

写选题时问自己：这个场景里，谁和谁的认知不在一个频道上？笑点就在那个频道差里。
